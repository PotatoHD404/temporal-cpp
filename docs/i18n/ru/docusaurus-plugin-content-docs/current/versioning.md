# Версионирование и патчинг

Воркфлоу — это детерминированная программа, выполнение которой восстанавливается
**реплеем его истории**. Когда воркер берёт в работу задачу воркфлоу, он не
возобновляется из сохранённого снимка ваших C++-объектов — он заново прогоняет
вашу функцию воркфлоу с самого начала, скармливая ей записанные события, и
сопоставляет испускаемые ею команды с событиями, уже находящимися в истории (см.
[Обнаружение недетерминизма](advanced.md#non-determinism-detection)). Именно это
делает воркфлоу долговечным, и это же делает *изменение кода* **выполняющегося**
воркфлоу опасным.

Эта страница — авторитетное руководство по эволюции кода воркфлоу без поломки
выполнений в процессе.

## Зачем версионирование {#why}

Предположим, v1 воркфлоу выполняет активность `A`, затем активность `B`:

```cpp
int OrderWorkflow(temporal::workflow::Context& ctx) {
  temporal::ActivityOptions o;
  o.start_to_close_timeout = std::chrono::seconds(10);
  const int a = ctx.ExecuteActivity<int>(o, "A").Get();
  const int b = ctx.ExecuteActivity<int>(o, "B", a).Get();
  return b;
}
```

Воркфlow стартует, выполняет `A`, записывает `ScheduleActivityTask{A}` /
`ActivityTaskCompleted{A}` в свою историю, а затем паркуется (ожидая `B`,
таймер, сигнал — всё, что охватывает более одной задачи воркфлоу). Пока он
запаркован, вы развёртываете новый воркер, в котором кто-то «улучшил» код, чтобы
вызывать `C` перед `B`:

```cpp
const int a = ctx.ExecuteActivity<int>(o, "A").Get();
const int c = ctx.ExecuteActivity<int>(o, "C").Get();   // НОВЫЙ первый шаг
const int b = ctx.ExecuteActivity<int>(o, "B", a + c).Get();
```

Когда запаркованное выполнение получает свою следующую задачу воркфлоу, новый
воркер реплеит с самого начала. Первое порождающее команду событие истории —
`ScheduleActivityTask{A}`, и код по-прежнему испускает `A` — нормально. Но
*второе* такое событие истории — `ScheduleActivityTask{B}`, тогда как новый код
теперь испускает `C`. Реплеемый поток команд расходится с историей:
**недетерминизм**. Задача проваливается согласно
[`WorkflowPanicPolicy`](#panic-policy), и в зависимости от политики воркфлоу либо
застревает (повторяется вечно против всё ещё сломанного кода), либо падает
сразу же.

Исправление состоит в том, чтобы никогда не мутировать путь, который уже прошла
существующая история. Вместо этого вы **ограничиваете новое поведение проверкой
версии**, чтобы старые истории продолжали идти по старому пути, а по новому шли
только новые выполнения. Именно для этого нужен `GetVersion`.

## `GetVersion`: патчинг на основе маркеров {#getversion}

```cpp
int GetVersion(const std::string& change_id, int min_supported, int max_supported);
```

`ctx.GetVersion` возвращает номер версии для именованного изменения. В **первый**
раз, когда воркфлоу доходит до вызова, он выбирает `max_supported`, записывает его
в историю как маркер и возвращает его. При каждом **реплее** он читает записанный
номер обратно из этого маркера и возвращает *его* — поэтому выполнение, которое
впервые запустилось на версии 1, всегда видит версию 1, даже после того как вы
повысите `max_supported` до 2 в коде.

Единственный особый случай — это история, записанная *до* того, как появился
вызов `GetVersion`: маркера для чтения нет, поэтому `GetVersion` возвращает
`temporal::workflow::kDefaultVersion` (который равен `-1`). Передача
`kDefaultVersion` как `min_supported` означает «я по-прежнему поддерживаю
неверсионированную, до-патчевую ветку».

```cpp
inline constexpr int kDefaultVersion = -1;  // temporal::workflow::kDefaultVersion
```

### Добавить шаг

Введите `C` между `A` и `B` только для новых выполнений, тогда как запаркованные
истории v1 продолжают идти прямо от `A` к `B`:

```cpp
int OrderWorkflow(temporal::workflow::Context& ctx) {
  temporal::ActivityOptions o;
  o.start_to_close_timeout = std::chrono::seconds(10);

  const int a = ctx.ExecuteActivity<int>(o, "A").Get();

  int extra = 0;
  const int v = ctx.GetVersion("add-step-C", temporal::workflow::kDefaultVersion, 1);
  if (v >= 1) {
    extra = ctx.ExecuteActivity<int>(o, "C").Get();  // C выполняют только новые выполнения
  }
  // до-патчевые выполнения: v == kDefaultVersion, extra остаётся 0, история без изменений

  return ctx.ExecuteActivity<int>(o, "B", a + extra).Get();
}
```

Выполнение, запущенное до этого развёртывания, реплеится с `v == kDefaultVersion`,
пропускает `C` и воспроизводит свой исходный поток команд `A`→`B` — детерминированно.
Выполнение, запущенное после развёртывания, записывает `add-step-C → 1` и
выполняет `C`.

### Изменить шаг

Замена одной активности другой имеет ту же форму — ветвитесь по версии вместо
правки на месте:

```cpp
const int v = ctx.GetVersion("swap-B-for-B2", temporal::workflow::kDefaultVersion, 1);
if (v == temporal::workflow::kDefaultVersion) {
  result = ctx.ExecuteActivity<int>(o, "B", a).Get();    // старые выполнения
} else {
  result = ctx.ExecuteActivity<int>(o, "B2", a).Get();   // новые выполнения
}
```

Это в точности та правка, которую — сделанную *без* ограждения — реплеер помечает
как недетерминизм (в тестовом наборе `ReplayFwV1` → `ReplayFwV2` меняет `AddOne`
на `Multiply`, и офлайн-реплеер это отлавливает — см. [Отлов поломок](#catching-breakage)).

### Внести несколько несовместимых изменений со временем

`max_supported` растёт по мере того, как вы продолжаете менять ту же область.
Каждое значение отображается на одну эпоху кода; старые истории закрепляются за
своим записанным номером:

```cpp
const int v = ctx.GetVersion("pricing", temporal::workflow::kDefaultVersion, 2);
if (v == temporal::workflow::kDefaultVersion) {
  price = legacy_price(ctx);          // до-патчевые выполнения
} else if (v == 1) {
  price = ctx.ExecuteActivity<int>(o, "PriceV1").Get();
} else {
  price = ctx.ExecuteActivity<int>(o, "PriceV2").Get();  // == 2
}
```

### Вывести из обращения старую версию

Как только **ни одно выполняющееся выполнение** больше не может находиться на
старой ветке — каждый воркфлоу, записавший меньшую версию (или неверсионированный
маркер), закрылся — вы можете отрезать эту ветку и поднять `min_supported`.
Используйте [запросы видимости](advanced.md#visibility-sessions)
(`client.CountWorkflows("WorkflowType = 'OrderWorkflow' AND ExecutionStatus = 'Running'")`)
или собственный учёт, чтобы убедиться, что ни одного не осталось.

Сначала отбросим ветку `kDefaultVersion`:

```cpp
// Было: GetVersion("pricing", kDefaultVersion, 2). До-патчевых выполнений не осталось,
// поэтому версия 1 теперь нижняя граница. min_supported == max_supported закрепляет всех за 2.
const int v = ctx.GetVersion("pricing", 1, 2);
if (v == 1) {
  price = ctx.ExecuteActivity<int>(o, "PriceV1").Get();
} else {
  price = ctx.ExecuteActivity<int>(o, "PriceV2").Get();
}
```

Когда `min_supported == max_supported`, каждое выполнение (новое и реплеемое)
разрешается в это единственное число, и в конце концов вы можете удалить вызов
целиком — но только после того, как ни одна история вообще не ссылается на change
id.

:::warning Никогда не меняйте аргументы существующего вызова `GetVersion` беспечно.
Понижение `min_supported` ниже версии, которую записала какая-то история, или
удаление ветки, через которую всё ещё реплеится выполняющееся выполнение, заново
вводит ровно тот недетерминизм, для предотвращения которого был добавлен
`GetVersion`. Выводите из обращения снизу вверх и только когда счётчики
выполняющихся равны нулю.
:::

## Недетерминированные значения {#non-deterministic-values}

Версионирование защищает *форму* потока команд воркфлоу. Другой источник
недетерминизма — чтение значения, которое отличается между исходным запуском и
реплеем: настенные часы, случайное число, свежесгенерированный UUID. Вызов
`std::chrono::system_clock::now()`, `rand()` или библиотеки UUID напрямую в
воркфлоу — это баг детерминизма: реплей вычисляет иное значение, чем подразумевает
история.

Фиксируйте такие значения через контекст, чтобы они записывались один раз и
реплеились дословно. Оба подробно описаны в разделе
[Расширенные возможности](advanced.md#sideeffect); вкратце:

- **`ctx.SideEffect<T>(fn)`** выполняет `fn` ровно один раз, записывает его
  результат в историю и возвращает записанное значение при каждом реплее:

  ```cpp
  std::string id = ctx.SideEffect<std::string>([] { return make_uuid(); });
  ```

- **`ctx.MutableSideEffect(id, fn[, equals])`** снабжён ключом `id` и записывает
  новый маркер, только когда значение *меняется* (по `operator==` или
  пользовательскому предикату); тип результата `R` **выводится из `fn`**, поэтому
  вы не передаёте его явно:

  ```cpp
  // Записывает маркер, только когда tier действительно отличается от прошлого вызова.
  std::string tier = ctx.MutableSideEffect("tier", [&] { return compute_tier(state); });
  ```

Ни один из них не заменяет [активность](concepts.md#activities): побочный эффект
должен быть чистым локальным вычислением без внешне наблюдаемого результата. Всё,
что касается внешнего мира, относится к активности, чей результат долговечно
записывается тем же способом.

## Отлов поломок {#catching-breakage}

Весь смысл этой страницы — внести изменения *до* того, как до них доберётся
выполняющийся воркфлоу. Инструмент для этого — офлайн-реплеер:

```cpp
void Worker::ReplayWorkflowHistory(const std::string& history_json);
```

Он реплеит записанную историю против вашего **текущего** кода воркфлоу, не
обращаясь к серверу, и бросает исключение, если реплеемые команды расходятся с
историей. Экспортируйте реальную историю с помощью
`WorkflowHandle::FetchHistoryJson()` (или `temporal workflow show -o json`),
храните её как фикстуру и реплейте:

```cpp
// 1) Зафиксируйте репрезентативную историю один раз (например, из только что запущенного воркфлоу):
std::string history_json = handle.FetchHistoryJson();

// 2) В юнит-тесте выполните реплей против кода, который вы собираетесь выкатить:
temporal::worker::Worker replayer(client, "replay");  // никогда не Start()-уется — без RPC
replayer.RegisterWorkflow("OrderWorkflow", OrderWorkflow);
replayer.ReplayWorkflowHistory(history_json);  // бросает исключение, если OrderWorkflow отклонился
```

Совместимая правка (всё, что корректно ограждено `GetVersion`) реплеится чисто;
несовместимая (переупорядоченная активность, заменённый тип активности, удалённый
таймер) бросает исключение. Подключите реплей фикстуры в CI, чтобы небезопасное
изменение проваливало сборку вместо продакшн-воркфлоу — см. [Тестирование → CI](testing.md#ci) и
[Реплей записанных историй](testing.md#replaying-recorded-histories).

### `WorkflowPanicPolicy` {#panic-policy}

Когда недетерминизм *всё же* доходит до выполняющегося воркера, происходящее
управляется `WorkerOptions::panic_policy` (полное рассмотрение в
[Обнаружение недетерминизма](advanced.md#non-determinism-detection)):

```cpp
temporal::WorkerOptions opts;
opts.panic_policy = temporal::WorkflowPanicPolicy::BlockWorkflow;  // по умолчанию
temporal::worker::Worker worker(client, "my-task-queue", opts);
```

- **`BlockWorkflow`** (по умолчанию) — провалить *задачу* воркфлоу. Сервер
  повторяет её, поэтому откат к совместимому воркеру восстанавливает выполнение без
  потери данных. Именно это делает плохое развёртывание переживаемым: исправьте код
  (или откатите), переразверните — задача выполнится успешно.
- **`FailWorkflow`** — провалить *выполнение* воркфлоу целиком. Терминально.

Предпочитайте значение по умолчанию в продакшне: недетерминированное развёртывание
паркует затронутые воркфлоу вместо того, чтобы убивать их, выигрывая вам время на
выкатку исправления.

## Worker Build-ID / версионирование {#worker-versioning}

`GetVersion` — это версионирование *на уровне кода*, оно живёт внутри одной функции
воркфлоу. На **уровне развёртывания** Temporal может также маршрутизировать задачи
по **build id воркера**, чтобы воркфлоу оставался закреплён за той версией воркера,
которая его запустила, позволяя выкатить новый бинарник без ограждений
`GetVersion`. SDK предоставляет клиентские RPC для этого; собственно закрепление
обеспечивается **на стороне сервера**, и соответствующие API, как правило, требуют
включённой динамической конфигурации на сервере, поэтому относитесь к методам ниже
как к управляющей поверхности, а не к готовой к работе из коробки возможности.

API **набора совместимости** по build-id (`Client`):

```cpp
// Добавить `build_id` как совершенно новый набор по умолчанию в очереди задач.
client.UpdateWorkerBuildIdCompatibility("my-task-queue", "v2");
// Повысить существующий набор (тот, что содержит `build_id`) до набора по умолчанию.
client.PromoteWorkerBuildIdSet("my-task-queue", "v1");
// Прочитать наборы обратно; каждый внутренний вектор — это один совместимый набор, последний id = по умолчанию.
std::vector<std::vector<std::string>> sets =
    client.GetWorkerBuildIdCompatibility("my-task-queue");
```

Более новый API **на основе правил** (правила назначения + перенаправления —
отличный от наборов выше):

```cpp
// Направить новые выполнения на build id (правило назначения в начале списка).
client.InsertWorkerAssignmentRule("my-task-queue", "v2");
// Перенаправить старый build id на более новый (постепенная выкатка).
client.AddWorkerRedirectRule("my-task-queue", /*source=*/"v1", /*target=*/"v2");
// Прочитать текущие правила + токен конфликта для следующего обновления.
temporal::client::WorkerVersioningRules rules =
    client.GetWorkerVersioningRules("my-task-queue");
```

И современный API **Worker Deployments**:

```cpp
std::vector<std::string> deployments = client.ListWorkerDeployments();
temporal::client::WorkerDeploymentDescription d =
    client.DescribeWorkerDeployment("my-deployment");
// Повысить build id до текущего (оптимистичная конкуренция через токен конфликта):
std::string token = client.SetWorkerDeploymentCurrentVersion(
    "my-deployment", "v2", d.conflict_token);
```

:::note Этим RPC нужна настройка на стороне сервера, чтобы делать что-либо полезное.
API build-id и правил требуют динамической конфигурации версионирования воркеров
(например, `frontend.workerVersioningRuleAPIs=true`), а API развёртываний требуют
`system.enableDeploymentVersions=true`. На стандартном dev-сервере некоторые из них
являются no-op или бросают исключение. Когда сомневаетесь, тянитесь к `GetVersion`
— он работает везде без серверной конфигурации.
:::

## Безопасное развёртывание {#deploying}

Как изменение доходит до выполняющихся воркфлоу на практике:

- **Sticky-выполнение.** Воркер держит горячую, в памяти («sticky») копию
  состояния воркфлоу и продолжает его оттуда для последующих задач — без реплея.
  Полный реплей (а значит, ваш новый код) происходит только когда этого кэша нет:
  воркер перезапустился, был переразвёрнут, запись была вытеснена или задача была
  направлена на *другой* воркер. Вы можете наблюдать это разделение по
  `worker.cache_hits()` против `worker.replays()`. Следствие: во время скользящего
  развёртывания **старые и новые воркеры работают бок о бок**, и данный воркфлоу
  может быть продолжен старым воркером на одной задаче и реплеен с нуля новым на
  следующей. Оба должны сходиться в потоке команд — что и есть ровно та причина,
  по которой огороженные изменения безопасны, а неогороженные — нет.

- **Запускайте старый и новый код вместе.** Из-за вышесказанного считайте любое
  развёртывание смешанным флотом на всё его время. Изменение, огороженное
  `GetVersion`, корректно при смешивании: старые выполнения разрешаются в старую
  версию на любом воркере, новые — в новую. Неогороженное несовместимое изменение —
  *нет*: оно «работает» лишь на тех воркерах, чей код случайно совпадает с каждой
  историей.

### Чек-лист безопасных изменений

1. **Добавление** активности, таймера, дочернего воркфлоу или любой новой команды
   в *существующий* путь кода → огородите его `ctx.GetVersion`.
2. **Удаление или переупорядочивание** существующей команды на пути, по которому
   шли старые истории → огородите старый и новый путь `ctx.GetVersion`; никогда не
   правьте на месте.
3. **Переименование** *строки типа* воркфлоу или активности — несовместимое
   изменение; относитесь к нему как к «изменить шаг».
4. Чтение **часов, случайности или UUID** напрямую → оберните в
   `ctx.SideEffect` / `ctx.MutableSideEffect`.
5. Изменение набора обработчиков **сигналов/запросов/обновлений воркфлоу или типов
   их аргументов** может оставить без обработки сообщения в процессе —
   версионируйте поведение обработчика тем же способом.
6. **Перед развёртыванием:** реплейте репрезентативные фикстуры историй с помощью
   `Worker::ReplayWorkflowHistory` в CI ([testing.md#ci](testing.md#ci)).
7. **Держите `panic_policy` на `BlockWorkflow`** в продакшне, чтобы пропущенный
   случай парковал, а не убивал затронутые выполнения.
8. **Выводите версию из обращения только** после того, как
   `CountWorkflows(... ExecutionStatus = 'Running')` подтвердит, что ни одно
   выполнение больше не может быть на старой ветке, затем поднимайте `min_supported`
   снизу вверх.

Изменения, которые *всегда* безопасны (версионирование не нужно): правка **тел**
активностей (активности не реплеятся — реплеятся лишь их записанные результаты),
