# Ключевые концепции

Если вы только знакомитесь с Temporal — здесь изложены все необходимые идеи. Если вы уже работали с другим Temporal SDK, соответствие один-к-одному сохраняется — отличается лишь C++-интерфейс. Эта страница — концептуальный справочник *что и почему*: каждая концепция получает короткое объяснение, C++-символ, в который она преобразуется, и ссылку на практическое руководство, показывающее её во всей полноте. Код здесь сведён к минимуму, необходимому лишь чтобы назвать тип.

:::note
Каждая концепция ниже зеркалит официальную концепцию Temporal. Задача SDK — заставить *критичную к детерминизму* часть работать нативно на C++; словарь (воркфлоу, активность, очередь задач, сигнал…) идентичен SDK для Go/Java/TypeScript.
:::

## Общая картина

**Клиент Temporal** общается с **сервером** Temporal («кластером»), чтобы запускать воркфлоу и взаимодействовать с ними. Сервер никогда не выполняет ваш код — он долговечно хранит **историю событий** (Event History) каждого воркфлоу и раздаёт **задачи**. Ваш процесс-**воркер** опрашивает **очередь задач**, выполняет код воркфлоу и активностей и отчитывается обратно. Сервер записывает результат как новые события. Этот цикл — *сервер хранит историю, воркер продвигает работу, сервер записывает результат* — и есть вся модель.

```
 Client  ──start/signal/query──▶  Server (history + task queues)  ◀──poll/respond──  Worker
                                          │                                            │
                                          └────────────── runs your ───────────────────┘
                                                       workflows & activities
```

Остальная часть страницы разворачивает каждый из этих блоков.

## Воркфлоу

**Воркфлоу** — это долговечный, отказоустойчивый оркестрационный код. Вы пишете его как обычную функцию:

```cpp
R MyWorkflow(temporal::workflow::Context& ctx, Args... args);
```

Важный момент: код воркфлоу должен быть **детерминированным**. Temporal сохраняет *историю* всего произошедшего (активность запланирована, таймер сработал, сигнал получен…) и перезапускает вашу функцию на этой истории, чтобы восстановить состояние после сбоя или при возобновлении на другом воркере. Поэтому воркфлоу не должен вызывать `rand()`, напрямую читать системное время, выполнять ввод-вывод или иным образом зависеть от чего-либо, не зафиксированного в истории. Вся оркестрация проходит через `workflow::Context`, который её и записывает.

Воркфлоу регистрируется через `Worker::RegisterWorkflow(name, fn)`; клиент запускает его через `Client::StartWorkflow(options, name, args...)`. Каждый запущенный экземпляр — это **выполнение воркфлоу** (Workflow Execution), идентифицируемое **идентификатором воркфлоу** (Workflow Id — вашим, в `StartWorkflowOptions::id`) плюс назначаемым сервером **Run Id**. Оба доступны внутри воркфлоу через `ctx.GetInfo()` (структура `WorkflowInfo`).

→ См. [Воркфлоу](/workflows/overview).

## Активности {#activities}

**Активность** — это обычная единица работы: функция, которая может делать что угодно (сетевые вызовы, диск, вычисления). Активности **не** подвергаются реплею — они выполняются единожды (с повторами) в реальном времени:

```cpp
R MyActivity(temporal::activity::Context& ctx, Args... args);
```

Воркфлоу планирует активность через `ctx.ExecuteActivity<R>(opts, name, args...)` и получает `Future<R>` с её результатом. Сервер гарантирует выполнение **как минимум один раз** (at-least-once): он берёт на себя повторы, тайм-ауты и повторную доставку, поэтому активность может выполниться более одного раза и там, где это важно, должна быть идемпотентной. Обычно активность завершается, когда функция возвращает результат, но она также может отложить завершение (`ctx.defer_completion()`) и быть завершена out-of-band позднее через `Client::CompleteActivity` / `FailActivity` — это полезно, когда результат зависит от внешнего колбэка.

Долгоработающие активности сообщают о прогрессе и живости через `activity::Context::RecordHeartbeat(...)`, который также показывает запрошенную сервером отмену (`ctx.IsCancelled()`).

→ См. [Воркфлоу](/workflows/overview) и [Тайм-ауты и повторы](/timeouts-and-retries).

## Воркеры, задачи и очереди задач

**Воркер** — это ваш процесс; он опрашивает **очереди задач** в ожидании работы. Существует два вида задач:

- **Задачи воркфлоу** продвигают воркфлоу вперёд: воркер запускает/возобновляет код воркфлоу и отвечает *командами* (запланировать активность, запустить таймер, завершить воркфлоу…).
- **Задачи активности** выполняют функцию активности и возвращают результат.

**Очередь задач** — это просто именованная очередь, которую сервер использует для маршрутизации работы воркерам. Никакой настройки не требуется — достаточно указать имя очереди в `Client::StartWorkflow` (`StartWorkflowOptions::task_queue`) и в конструкторе `Worker`; любой воркер, опрашивающий это имя, может взять задачу. По умолчанию активности и дочерние воркфлоу наследуют очередь задач своего воркфлоу; переопределите её на отдельный вызов через `ActivityOptions::task_queue` / `ChildWorkflowOptions::task_queue`, чтобы направить работу другому пулу воркеров.

Вы регистрируете функции воркфлоу и активностей в объекте `Worker` и вызываете `Start()` (неблокирующий) или `Run()` (блокирует до SIGINT / `Stop()`). Строковое имя, передаваемое в `RegisterWorkflow` / `RegisterActivity`, — это то же имя, по которому запускают воркфлоу или планируют активность. Чтобы в этих именах нельзя было ошибиться, привяжите имя к его функции типизированным дескриптором — `TEMPORAL_ACTIVITY(fn)` вместе с `worker.Register(fn_activity)` и `ctx.ExecuteActivity(opts, fn_activity, args...)` — и тогда имя, типы аргументов и тип результата проверяются на этапе компиляции.

→ См. [Клиент и воркер](/client-and-worker).

## Клиент Temporal

`temporal::client::Client` — это соединение с frontend-сервисом Temporal (дёшево копируется, поскольку использует общий gRPC-канал). Это точка входа для всего *вне* воркфлоу:

- запустить воркфлоу (`StartWorkflow`) или signal-with-start (`SignalWithStartWorkflow`);
- получить дескриптор существующего воркфлоу (`GetHandle`), а затем вызвать `Signal`, `Query`, `Update`, `Cancel`, `Terminate` или дождаться его результата через `Result<R>()`;
- выполнять запросы видимости, управлять расписаниями, поисковыми атрибутами, Nexus-эндпоинтами и пакетными операциями (рассмотрено ниже).

`Client::Connect(ClientOptions{...})` открывает соединение. `WorkflowHandle` — это представление одного выполнения; `Client` — представление на уровне кластера.

→ См. [Клиент и воркер](/client-and-worker).

## История событий и команды

Сервер ведёт **историю событий** (Event History) с дозаписью в конец для каждого выполнения воркфлоу — долговечный источник истины. Воркер никогда не изменяет историю напрямую. Вместо этого, выполняя задачу воркфлоу, он испускает **команды** (намерение: *запланировать эту активность*, *запустить этот таймер*, *завершить воркфлоу*), а сервер валидирует их и записывает соответствующие **события** (*ActivityTaskScheduled*, *TimerStarted*, *WorkflowExecutionCompleted*, …). Более поздние события (*ActivityTaskCompleted*, *TimerFired*, *WorkflowExecutionSignaled*) записывают, что произошло дальше, и порождают следующую задачу воркфлоу.

В этом SDK вы никогда не пишете команды вручную — вызов `ctx.ExecuteActivity`, `ctx.NewTimer`, `ctx.ExecuteChildWorkflow` или возврат из воркфлоу *и есть* то, что их испускает. Сырую историю воркфлоу можно получить как Temporal JSON через `WorkflowHandle::FetchHistoryJson()` (удобно для тестирования реплея).

→ См. [Архитектура](/architecture).

## Детерминизм и реплей

Поскольку история является источником истины, движок восстанавливает состояние воркфлоу в памяти, **реплицируя** его код на записанной истории: он перезапускает вашу функцию, и каждый оркестрационный вызов возвращает значение, уже записанное в истории, вместо повторного выполнения работы. Чтобы это каждый раз порождало одни и те же команды, **код воркфлоу должен быть детерминированным** — одна и та же история на входе, одни и те же команды на выходе. Всё недетерминированное (системное время, ГСЧ, ввод-вывод, порядок обхода `map`, чтение внешнего состояния) должно проходить через `Context`, чтобы быть записанным:

| Вместо…                  | Используйте…                                  |
| ------------------------ | --------------------------------------------- |
| `std::this_thread::sleep`| `ctx.Sleep(d)` / `ctx.NewTimer(d)`            |
| `rand()`, новый UUID     | `ctx.SideEffect<R>(fn)`                        |
| чтения системного времени| `ctx.SideEffect<R>(fn)` (зафиксируйте время однократно)|
| прямого сетевого/дискового ввода-вывода | **активность**                  |

Чтобы реплей был дешёвым, движок держит корутину выполняющегося воркфлоу резидентной в **sticky-кэше** между задачами; только при промахе кэша (первая задача или после вытеснения) он реплеит из полной истории. См. *Как движок выполняет ваш воркфлоу* ниже.

Если воркер когда-либо порождает команды, которые **расходятся** с записанной историей (например, вы несовместимо изменили код воркфлоу), — это **ошибка недетерминизма**; о том, как обработать её безопасно, см. [Безопасность детерминизма](#determinism-safety-panic-policy--deadlock-detection) и [Версионирование](#versioning).

→ См. [Архитектура](/architecture) и [Версионирование](/versioning).

## Таймеры

**Таймер** — это долговечный, безопасный для реплея способ для воркфлоу ждать. `ctx.NewTimer(duration)` возвращает `Future<void>`, который завершается, когда таймер срабатывает; `ctx.Sleep(duration)` — это сокращение (`NewTimer(...).Get()`):

```cpp
ctx.Sleep(std::chrono::hours(24));            // долговечно — переживает перезапуски воркера
auto t = ctx.NewTimer(std::chrono::minutes(5));  // состязайте его с другой работой в Selector
```

Таймер записывается в историю, поэтому воркфлоу может «спать месяц» сквозь сбои, деплои и переезды воркера. Отмените ожидающий таймер через `Future::Cancel()`. Никогда не используйте настоящий OS-sleep в коде воркфлоу — он не долговечен и ломает детерминизм.

## Сигналы, запросы и обновления

Выполняющийся воркфлоу может обмениваться сообщениями с внешним миром, не завершаясь:

- **Сигнал** асинхронно доставляет данные *в* воркфлоу (`ctx.GetSignalChannel<T>(name)` на стороне воркфлоу, возвращающий `ReceiveChannel<T>`; `handle.Signal(name, value)` со стороны клиента). Он записывается в историю и долговечен. `SignalWithStartWorkflow` доставляет сигнал и запускает воркфлоу, если он ещё не выполняется, атомарно.
- **Запрос** (query) синхронно читает состояние воркфлоу и **не** должен его изменять или планировать работу (`ctx.SetQueryHandler(name, fn)`; `handle.Query<R>(name, args...)`). Запросы выполняются на живом кэшированном состоянии и никогда не появляются в истории.
- **Обновление** (update) — это синхронный вызов, который *может* изменять состояние и **записывается** (`ctx.SetUpdateHandler(name, fn)`; `handle.Update<R>(name, args...)`). У него может быть read-only **валидатор** (трёхаргументная перегрузка `SetUpdateHandler`), отклоняющий вызов до того, как он будет принят, — тогда в историю ничего не пишется, а обработчик не запускается.

Как и в случае с активностями, на каждый из них можно сослаться типизированным дескриптором — `SignalRef<T>`, `QueryRef<R>`, `UpdateRef<R>` — так что типы полезной нагрузки и результата проверяются, а не задаются заново строкой плюс явным шаблонным аргументом.

→ См. [Обмен сообщениями](/workflows/messaging).

## Дочерние воркфлоу

Воркфлоу может запустить другой воркфлоу как **дочерний**, дав ему собственную историю, повторы и идентичность, сохраняя при этом связь «родитель/потомок»:

```cpp
auto fut = ctx.ExecuteChildWorkflow<R>(temporal::ChildWorkflowOptions{...}, "ChildType", args...);
R result = fut.Get();
```

Прибегайте к дочернему воркфлоу (а не к активности), когда подзадача сама по себе долгоработающая, оркестрирует собственные активности или заслуживает отдельной истории по соображениям её размера или разделения ответственности. `ChildWorkflowOptions::parent_close_policy` (`Terminate` / `Abandon` / `RequestCancel`) решает, что произойдёт с ещё выполняющимся дочерним воркфлоу, когда родитель завершается.

→ См. [Композиция](/workflows/composition).

## Continue-As-New

История не может расти вечно — воркфлоу, выполняющий бесконечный цикл (долгоживущая «сущность» или цикл опроса), накопил бы неограниченное число событий. **Continue-As-New** атомарно завершает текущий запуск и стартует новый с тем же идентификатором воркфлоу и совершенно новой, пустой историей, перенося только то состояние, которое вы передаёте как новый вход:

```cpp
ctx.ContinueAsNew("MyWorkflow", carriedOverState);  // [[noreturn]] — никогда не возвращает
```

Это стандартный паттерн для периодических или бесконечных воркфлоу. Сначала вычерпайте ожидающие сигналы; вызов терминальный, поэтому всё после него недостижимо.

## Политики повторов и тайм-ауты

Активности (и воркфлоу, где это поддерживается) автоматически повторяются при сбое согласно `RetryPolicy` (`initial_interval`, `backoff_coefficient`, `maximum_interval`, `maximum_attempts`, `non_retryable_error_types`). Выброс `ApplicationError` с `non_retryable = true` (или соответствующим типом) немедленно прекращает повторы. Тайм-ауты ограничивают каждую фазу активности — `schedule_to_close_timeout`, `schedule_to_start_timeout`, `start_to_close_timeout`, `heartbeat_timeout` в `ActivityOptions` — и сам воркфлоу (`execution_timeout` / `run_timeout` / `task_timeout` в `StartWorkflowOptions`).

→ См. [Тайм-ауты и повторы](/timeouts-and-retries).

## Расписания

**Расписание** (Schedule) запускает воркфлоу по периодической спецификации без постоянно работающего драйвера. Создайте его через `Client::CreateSchedule(schedule_id, ScheduleOptions{...})`, где `ScheduleOptions` принимает `interval` и/или `cron_expressions` плюс `workflow_type` / `task_queue` для запуска. Расписания можно описывать, обновлять, запускать по требованию, ставить на паузу/снимать с паузы, перечислять и удалять через `Client`.

→ См. [Расписания](/schedules).

## Видимость и поисковые атрибуты

**Видимость** (Visibility) — это возможность *перечислять и подсчитывать* выполнения воркфлоу по их свойствам. `Client::ListWorkflows(query)` и `Client::CountWorkflows(query)` принимают list-фильтр Temporal (например, `"WorkflowType = 'Foo' AND ExecutionStatus = 'Running'"`) и постранично перебирают совпадения. Фильтр может ссылаться на **поисковые атрибуты** (search attributes) — индексированные поля ключ/значение, прикреплённые к выполнению.

Задавайте поисковые атрибуты при запуске (`StartWorkflowOptions::search_attributes`) или изнутри выполняющегося воркфлоу (`ctx.UpsertSearchAttributes(...)`). Стройте типизированные значения с помощью хелперов `temporal::sa::` (`sa::Keyword`, `sa::Text`, `sa::Int`, `sa::Double`, `sa::Bool`, `sa::KeywordList`). Пользовательские атрибуты должны быть сначала зарегистрированы в неймспейсе — управляйте ими через `Client::AddSearchAttributes` / `ListSearchAttributes` / `RemoveSearchAttributes`. Неиндексированные метаданные, которые вы хотите получать обратно только при `Describe` (но не искать), помещаются в `memo`.

→ См. [Клиент и воркер](/client-and-worker).

## Неймспейсы

**Неймспейс** (namespace) — это единица изоляции в кластере: воркфлоу, очереди задач и поисковые атрибуты одного неймспейса невидимы для другого. Клиент подключается ровно к одному неймспейсу, задаваемому через `ClientOptions::ns` (по умолчанию `"default"`); каждый его вызов — запуск, сигнал, перечисление, расписание — работает в пределах этого неймспейса. Используйте отдельные неймспейсы, чтобы изолировать команды, окружения (dev/staging/prod) или тенантов. Администрирование неймспейсов (например, `Client::DeleteNamespace`) также находится на `Client`.

## Nexus

**Nexus** — это способ Temporal вызывать код за границами команд или неймспейсов через чистый, слабосвязанный API. Воркер регистрирует обработчик операции (`worker.RegisterNexusOperation(service, operation, fn)`), эндпоинт именует целевую очередь задач (`client.CreateNexusEndpoint(name, task_queue)`), а воркфлоу вызывает её через `ctx.ExecuteNexusOperation<R, Arg>(endpoint, service, operation, input).Get()`. В отличие от активности, Nexus-операция принимает **одно** входное значение и возвращает **одно** значение (по одному `Payload` на каждое). Эндпоинты создаются, перечисляются и описываются через `Client` (`CreateNexusEndpoint` / `GetNexusEndpoint` / `ListNexusEndpoints`).

→ См. [Продвинутые возможности](/advanced).

## Версионирование {#versioning}

Когда вы изменяете уже развёрнутый код воркфлоу, изменение не должно ломать выполнения «на лету», которые реплеят на историях, записанных старым кодом. `ctx.GetVersion(change_id, min, max)` записывает маркер версии при первом запуске и возвращает его при каждом реплее, так что воркфлоу может безопасно ветвиться между старым и новым кодом (возвращая `kDefaultVersion`, `-1`, для историй, записанных до того, как вызов появился). API версионирования Worker Build-Id и Worker Deployment на `Client` маршрутизируют новые выполнения на новые сборки воркеров на уровне всего флота.

→ См. [Версионирование](/versioning).

## Конвертация данных

Каждое значение, пересекающее границу — аргументы/результаты воркфлоу, аргументы/результаты активностей, payload-ы сигналов/запросов/обновлений, memo, поисковые атрибуты — сериализуется в `Payload` (непрозрачные байты плюс метаданные; ключ `encoding`, например `json/plain`, выбирает декодер) с помощью **DataConverter**. Конвертер по умолчанию кодирует как JSON; вы можете предоставить собственный (`ClientOptions::data_converter`), чтобы изменить сериализацию или добавить слой **кодека** (codec) для сжатия либо сквозного шифрования байтов payload. Типизированные API (`ExecuteActivity<R>`, `Query<R>`, `Signal(name, value)`, …) вызывают конвертер за вас, так что код приложения оперирует C++-типами, а не payload-ами.

→ См. [Конвертация данных](/data-conversion).

## Future и блокирование

Оркестрационные вызовы возвращают `Future<R>`:

```cpp
auto future = ctx.ExecuteActivity<std::string>(opts, "Act", arg);
std::string result = future.Get();   // блокирует воркфлоу, пока активность не завершится
```

`Future::Get()` выглядит как обычный блокирующий вызов, однако под капотом он **приостанавливает** воркфлоу (воркер отвечает серверу и ожидает следующей задачи), а затем возобновляет его, когда результат появляется в истории.

Если вы предпочитаете корутины C++20, воркфлоу может возвращать `workflow::workflow_task<R>` и применять `co_await` к future вместо `.Get()` — `co_await ctx.ExecuteActivity<R>(...)` в точности эквивалентно `.Get()` (те же команды, тот же реплей). Чтобы ждать несколько случаев сразу — скажем, сигнал *или* таймаут — используйте `workflow::Selector`, который продолжает работу, когда первый из его случаев future/канала становится готов (`AddFuture` / `AddReceive` / `AddDefault` / `Select`).

## Тестирование и реплей

Два средства позволяют тестировать воркфлоу без продакшен-сервера:

- **Реплей** — прогоняет записанную историю (`Worker::ReplayWorkflowHistory`) обратно через код вашего воркфлоу вообще без сервера. Если команды кода расходятся с записью, реплей бросает исключение — это ловит недетерминированное изменение прежде, чем оно попадёт в продакшен.
- **Проматывание времени** — `testing::TestWorkflowEnvironment` подключается к `temporal-test-server`, который быстро проматывает ожидающие таймеры, так что воркфлоу, спящий «днями», завершается за миллисекунды.

→ См. [Тестирование](/testing).

## Как движок выполняет ваш воркфлоу

Этот SDK реализует критически важную часть — обеспечение детерминизма — нативно. Вкратце:

1. Тело воркфлоу выполняется на **стекозависимой корутине** (кооперативном потоке-файбере). Ожидание неразрешённого future *уступает* управление корутине, сохраняя весь её стек и локальные переменные.
2. **Sticky-кэш** удерживает корутину выполняющегося воркфлоу в памяти между задачами. Задачи продолжения несут лишь *новые* события истории, которые движок применяет к живым future перед возобновлением — без полного реплея. `WorkerOptions::max_cached_workflows` ограничивает кэш (вытеснение по LRU); `Worker::cache_hits()` / `replays()` показывают число попаданий/промахов.
3. При промахе кэша (первая задача или после вытеснения) движок выполняет реплей из полной истории для восстановления состояния, после чего помещает воркфлоу в кэш.

Поскольку корутина сохраняет состояние живым между приостановками, **запросы** (query) и **селекторы** могут читать актуальное состояние воркфлоу — по той же причине, по которой официальные SDK держат воркфлоу резидентными в памяти.

## Безопасность детерминизма, политика паник и обнаружение дедлоков {#determinism-safety-panic-policy--deadlock-detection}

Функции обеспечения детерминизма тоже реализованы:

- **Обнаружение недетерминизма** — воспроизведённые команды, расходящиеся с записанной историей, проваливают задачу воркфлоу согласно `WorkflowPanicPolicy` воркера (`BlockWorkflow`, безопасное значение по умолчанию, проваливает только *задачу*, так что исправленная сборка может восстановиться; `FailWorkflow` проваливает *выполнение*).
- **Версионирование** (`ctx.GetVersion`) для безопасного выкатывания изменений в коде воркфлоу (см. [Версионирование](#versioning)).
- **Побочные эффекты** (`ctx.SideEffect` / `ctx.MutableSideEffect`) для записи разового недетерминированного значения в историю.
- **Обнаружение дедлоков** — `WorkerOptions::deadlock_detection_timeout` ограничивает, как долго может выполняться одно возобновление задачи воркфлоу; воркфлоу, заблокировавшийся на настоящем OS-вызове или зациклившийся навсегда, репортится, а его задача прерывается, тогда как воркер продолжает обслуживать другие воркфлоу.

Принятые **обновления** (update) при полном реплее также повторно применяются в записанном чередовании.

→ См. [Продакшн](/production).
