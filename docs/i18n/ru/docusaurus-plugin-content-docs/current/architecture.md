# Архитектура

`temporal-cpp-sdk` — это **нативный** C++ SDK Temporal: он напрямую взаимодействует с frontend-сервером Temporal через gRPC и самостоятельно реализует движок реплея воркфлоу, критичный к детерминизму, отражая структуру официального [Go SDK](https://github.com/temporalio/sdk-go). Он **не** встраивает Rust-ядро [`sdk-core`](https://github.com/temporalio/sdk-core), на котором построены SDK для Python/TypeScript/.NET/Ruby.

## Слоистая структура (зеркалит Go SDK)

```
include/temporal/            Public API (proto-free)
  client/  worker/  workflow/  activity/  converter/
  common/  interceptor/  log/  testing/
src/
  client/  worker/                 Thin public-facing implementations
  converter/  log/                 Data conversion, structured logging
  internal/                        The engine (never installed, never in public headers):
    grpc_client          WorkflowService stub wrapper
    coroutine            Stackful thread-fiber coroutine
    workflow_task_handler  Replay engine + sticky cache + WorkflowRunner (the crux)
    activity_task_handler  Activity execution + heartbeating
    nexus_task_handler   Nexus operation execution
    worker_impl          Poller threads + lifecycle
    determinism          Non-determinism detection (command-vs-history matching)
    lru_cache            Bounded sticky-cache LRU
    proto_util           Payload/Failure/Duration <-> protobuf, ids
```

Разделение public/`internal` соответствует пакетам `client`/`worker`/`workflow` в Go поверх его большого движка в `internal/`. Жёсткое правило: **публичные заголовки никогда не включают protobuf** — вся gRPC/proto-поверхность находится исключительно в `src/internal/`.

## Проблема детерминизма

Воркфлоу Temporal должен быть **детерминированным и воспроизводимым**: его история событий (активность запланирована/завершена, таймер запущен/сработал, …) является источником истины, и SDK обязан восстановить точное состояние воркфлоу в памяти, реплицируя эту историю. Каждый оркестрационный вызов, который делает воркфлоу, перехватывается, превращается в *команду* для сервера и впоследствии сопоставляется обратно с получившимися событиями истории. Сделать это правильно — наиболее сложная, нативная часть любого SDK.

## Движок реплея воркфлоу

Движок живёт в `src/internal/workflow_task_handler.cpp`. Холодный старт (или промах sticky-кэша) запускает путь **реплея**: сервер присылает полную историю событий вместе с задачей воркфлоу, и мы заново выполняем функцию воркфлоу с самого начала на этой истории.

**1. Предсканирование истории** (`ScanHistory`). Один проход по событиям строит:

- вход воркфлоу (из `WorkflowExecutionStarted`);
- исходы по каждой активности (`scheduled`, а также `completed`+результат / `failed`+ошибка / `timed_out`), сопоставляя события завершения с их планированием через несомый ими `scheduled_event_id`;
- исходы по каждому таймеру, дочернему воркфлоу и Nexus-операции (`started`/`fired`, initiated/completed);
- доставленные сигналы, флаг запроса отмены и записанные маркеры (побочные эффекты, версии).

**2. Детерминированные идентификаторы операций.** `WorkflowRunner` назначает каждому вызову `ExecuteActivity` следующий целочисленный порядковый номер как его `activity_id` (таймеры, дочерние воркфлоу и Nexus-операции — аналогично, каждый со своим префиксом). Поскольку воркфлоу детерминирован, *тот же* вызов при более позднем реплее получает *тот же* id, поэтому он сходится с событием истории из предыдущей попытки.

**3. Выполнение.** Пользовательская функция воркфлоу выполняется с `workflow::Context`, опирающимся на runner:

- `ExecuteActivity(seq)`: если `activity_id == seq` уже есть в истории → разрешить возвращаемый `Future` из записанного исхода и испустить **ноль** команд (реплей). Иначе → испустить команду `ScheduleActivityTask` и вернуть *ожидающий* future (первое выполнение).
- `Future::Get()`: если future готов, вернуть декодированное значение (или выбросить `ActivityError`). Если он ожидающий, **уступить** управление корутины, чтобы припарковать воркфлоу (см. ниже).
- Нормальный возврат функции порождает команду `CompleteWorkflowExecution`; `ApplicationError`/`std::exception` порождает `FailWorkflowExecution`; запрос `ContinueAsNew` порождает `ContinueAsNewWorkflowExecution`; парковка порождает только команды планирования, собранные на данный момент.

**4. Ответ.** Все испущенные команды (сначала планирования/таймеры, затем любая терминальная команда) возвращаются через `RespondWorkflowTaskCompleted`.

Это в точности та модель **активация → команды**, которую используют SDK на основе core: производное из истории состояние играет роль активации, а испущенные команды — роль завершения, — только здесь это вычисляется нативно на C++, а не средствами `sdk-core`.

### Разобранный пример (`GreetWorkflow`)

| Задача | Увиденная история | Что делает runner | Отправленные команды |
|------|--------------|-------------|---------------|
| 1 | `WorkflowExecutionStarted` | `ExecuteActivity("0")` нет в истории → запланировать; `Get()` ожидает → **парковка** | `ScheduleActivityTask(id=0)` |
| — | *(воркер активности выполняет `ComposeGreeting`, сервер записывает `ActivityTaskCompleted`)* | | |
| 2 | …`ActivityTaskScheduled(0)`, `ActivityTaskCompleted`→"Hello, Temporal!" | `ExecuteActivity("0")` разрешён → `Get()` возвращает; функция возвращает | `CompleteWorkflowExecution(result)` |

## Модель приостановки: диспетчер стекозависимых корутин

Тело воркфлоу выполняется на **стекозависимой корутине** (`src/internal/coroutine.h`) — кооперативном потоке-файбере, управляемом строгим пошаговым протоколом передачи управления (мьютекс + условная переменная), при котором диспетчер и поток воркфлоу никогда не работают одновременно; поэтому разделяемое ими состояние не требует никакой дополнительной блокировки. Это C++-аналог диспетчера на основе горутин, который использует Go SDK.

Ожидание неразрешённого future или сигнала **уступает** управление корутине — сохраняя весь стек воркфлоу и локальные переменные — вместо его раскрутки. При возобновлении воркфлоу выполняется до следующей точки приостановки; его живое состояние затем остаётся доступным для чтения запросами (query) и для выбора селекторами. Завершение (вытеснение из кэша или завершение воркфлоу) выбрасывает `CoroutineAbort` в приостановленный фрейм, чтобы корректно его раскрутить.

:::note
`CoroutineAbort` намеренно **не** наследуется от `std::exception`, поэтому `catch (const std::exception&)` воркфлоу не может его проглотить. Пользовательский код никогда не должен делать `catch (...)` так, чтобы это исключение отбрасывалось.
:::

## Sticky-кэш

Обработчик держит корутину каждого выполняющегося воркфлоу живой между задачами, с ключом по run id (`src/internal/workflow_task_handler.cpp`). Воркер объявляет уникальную **sticky-очередь задач** в каждом `RespondWorkflowTaskCompleted` и опрашивает её вторым поллером, поэтому сервер доставляет задачи продолжения, несущие только **инкрементальную** историю:

- **Продолжение** (id первого события задачи равен `cached.last_event_id + 1`): применить только новые события к живым future / состоянию сигналов (`ApplyEvents`) и вызвать `Resume` для кэшированной корутины — **без полного реплея**. Завершения активностей соотносятся со своим future через постоянную карту `scheduled_event_id → activity_id`.
- **Реплей** (первая задача или история не продолжает кэш): восстановить из полной истории путём предсканирования, выполнить до текущей точки приостановки и закэшировать runner.
- **Промах sticky-кэша** (инкрементальная история, но ничего не кэшировано, например после вытеснения): ответить `RESET_STICKY_TASK_QUEUE`, чтобы сервер переслал полную историю на обычной очереди.

Поскольку `Block`/`Park` написаны как циклы с повторной проверкой, один и тот же код корутины обслуживает оба пути.

Кэш представляет собой **ограниченный LRU** (`src/internal/lru_cache.h`): `WorkerOptions::max_cached_workflows` ограничивает резидентный набор (`0` = без ограничения), и при заполнении вытесняется наименее недавно использованный runner — его следующая задача идёт по пути `RESET_STICKY_TASK_QUEUE` и реплеит из полной истории. Завершённые воркфлоу вытесняются немедленно. `Worker::cache_hits()` / `Worker::replays()` показывают, сколько задач пошло по пути продолжения против реплея.

## Активации → команды

Каждая задача передаёт производное из истории состояние выполняющемуся воркфлоу, который порождает команды (`ScheduleActivityTask`, `StartTimer`, `StartChildWorkflowExecution`, `ScheduleNexusOperation`, `UpsertWorkflowSearchAttributes`, `RecordMarker`, `CompleteWorkflowExecution`, `ContinueAsNewWorkflowExecution`, …). Обновления (update) передаются как протокольные сообщения (`Acceptance` / `Response`), а не команды. Это та же модель активация→команда, которую используют SDK на основе core, только здесь реализованная нативно на C++.

### Обнаружение недетерминизма

В отличие от исходного движка, путь реплея теперь **валидирует** себя сам. `src/internal/determinism.h` захватывает упорядоченный поток команд, которые воркфлоу *порождает* во время реплея, и упорядоченный поток порождающих команды событий, которые *записала* история, а затем попарно сопоставляет их (`MatchReplayCommands` / `CommandMatchesEvent`, зеркаля `matchReplayWithHistory` / `isCommandMatchEvent` из Go SDK). История авторитетна: активности и дочерние воркфлоу ключуются по id + имени типа; таймеры, маркеры и Nexus-операции — по id; терминальные команды сопоставляются только по виду. Первое расхождение поднимает ошибку недетерминизма `[TMPRL1100]` и проваливает задачу воркфлоу, вместо того чтобы повредить состояние. Воркфлоу всё ещё может испускать *дополнительные* завершающие команды — это реальное продвижение вперёд за пределами воспроизведённой истории.

## Повторное применение обновлений при реплее {#replay-re-application-of-updates}

При sticky-**продолжении** движок просто применяет новые события к живым future и возобновляет работу, поэтому состояние, изменённое обновлением, уже находится в памяти. Реплей **с нуля** (холодный старт, вытеснение из sticky-кэша или `Worker::ReplayWorkflowHistory`) сложнее: предсканирование заранее разрешает каждый известный из истории future, поэтому одно наивное возобновление обогнало бы тело воркфлоу до завершения *прежде*, чем перезапустится хоть один принятый обработчик обновления, — и тело никогда не увидело бы состояние, заданное обновлением.

Чтобы воссоздать живое чередование, предсканирование разбивает историю на **первом принятом обновлении**. Всё, что до него, ускоренно проигрывается начальным возобновлением как обычно. Всё, что *после* него, — каждое разрешение future и доставленный сигнал — **откладывается** в `post_update_timeline` вместо предварительного применения. Затем движок проигрывает эту хронологию в **порядке истории**, вставляя принятые обновления обратно на их записанные позиции: каждое обновление перезапускает свой обработчик, каждое отложенное событие разрешает future или доставляет сигнал, а возобновление после каждого шага позволяет телу продвинуться до следующей точки парковки. Состояние, изменённое обновлением, — флаг, который тело ожидает через `Await`, счётчик, который завершает более поздний сигнал, — восстанавливается ровно таким, каким оно было вживую.

Поскольку испускаемые команды защищены от повторного срабатывания, перезапуск обработчика или повторное разрешение future никогда не перепланируют работу; они лишь продвигают тело и порядок порождаемых команд, с которым сравнивает проверка детерминизма. **Маркеры / побочные эффекты *не* откладываются** — они потребляются в порядке вызова, независимо от тайминга событий. Для распространённого воркфлоу без обновлений хронология пуста, поэтому реплей остаётся ускоренной перемоткой одним возобновлением, без изменений.

## Прерывание при дедлоке

Код воркфлоу никогда не должен блокироваться. Диспетчер возобновляет каждую корутину под необязательным ограничением на одно возобновление (`WorkerOptions::deadlock_detection_timeout`): возобновление выполняется через `Coroutine::ResumeFor(timeout)`, и задача, не уступившая управление в его пределах, помечает runner как заблокированный (deadlocked). Тогда обработчик **прерывает выполнение** — он испускает метрику `temporal_workflow_task_deadlock`, логирует ошибку, удаляет запуск из кэша и завершает *задачу* воркфлоу с ошибкой (причина `WorkflowDeadlock`), чтобы сервер перепланировал её.

Сорвавшуюся корутину нельзя безопасно прервать, поэтому она **покидается**: она продолжает работать на собственном отсоединённом потоке, а её runner перемещается в список утечек, живущий всё время процесса (`g_leaked_runners`), чтобы этот поток никогда не разыменовывал освобождённую память. Поток поллера немедленно возвращается и продолжает обслуживать другие воркфлоу. Это тот же компромисс, на который идёт Go SDK (горутина заблокированного воркфлоу утекает). Операционный взгляд см. в [Запуск в продакшне](/production#deadlock-detection).

## Сигналы и отмена

Оба приходят как события истории (`WorkflowExecutionSignaled`, `WorkflowExecutionCancelRequested`), поэтому вписываются в модель реплея:

- **Сигналы** — `ScanHistory` собирает каждое `WorkflowExecutionSignaled` в упорядоченный список по имени. `ctx.GetSignalChannel<T>(name)` возвращает `ReceiveChannel`, чей `Receive()` потребляет следующий сигнал через детерминированный курсор по имени (сбрасывается при каждом реплее) и декодирует его в `T`; если ни одного не осталось, он паркуется (уступает управление корутины) до следующего события. Поскольку реплей детерминирован, N-й `Receive()` всегда возвращает N-й сигнал — буферизация и порядок соответствуют семантике Temporal. `ReceiveAsync()` — неблокирующий вариант.
- **Отмена** — `WorkflowExecutionCancelRequested` устанавливает флаг, доступный как `ctx.IsCancelled()`, а `ctx.AwaitCancellation()` возвращает future, который можно состязать в `Selector` (канонический паттерн «работа ИЛИ отмена»). Воркфлоу сам выбирает, как реагировать (завершиться, прибраться); запрос отмены планирует задачу воркфлоу, так что воркфлоу, припаркованный на `Receive()` сигнала, перезапускается и может увидеть флаг. `ctx.CancelExternalWorkflow` / `ctx.SignalExternalWorkflow` достают другие выполнения по id. Полную модель см. в [Отмена](/cancellation).

## Запросы, обновления и селекторы

Они опираются на то, что корутина сохраняет живое состояние воркфлоу сквозь приостановку:

- **Запросы** — `ctx.SetQueryHandler(name, fn)` регистрирует read-only обработчик (перерегистрируется при каждом реплее). Когда приходит запрос, воркфлоу реплеится до текущей точки приостановки, и обработчик вызывается на живом, припаркованном состоянии, отвечая через `RespondQueryTaskCompleted` (или через `query_results` для запросов, прикреплённых к задаче воркфлоу). Клиентская сторона — `WorkflowHandle::Query<R>(type, args...)`.
- **Обновления** — `ctx.SetUpdateHandler(name, fn)` регистрирует обработчик, который может изменять состояние; трёхаргументная перегрузка добавляет read-only **валидатор**, который запускается первым и, выбрасывая исключение, отклоняет обновление до принятия (в историю ничего не пишется, обработчик не запускается). Обновление приходит как протокольное сообщение в задаче воркфлоу; обработчик выполняется на живом состоянии, и воркер отвечает протокольными сообщениями `Acceptance` + `Response` (повторяя тела из Go SDK, включая `accepted_request_sequencing_event_id`). Клиентская сторона — `WorkflowHandle::Update<R>(name, args...)`. Обновления применяются на живом (sticky) пути, а при реплее с нуля повторно применяются в порядке истории (см. [Повторное применение обновлений при реплее](#replay-re-application-of-updates)).
- **Селекторы** — `workflow::Selector` ожидает несколько future и продолжает выполнение, когда любой из них готов (канонический паттерн «активность ИЛИ таймаут»), запуская обработчик совпавшего случая; `Select()` паркуется через корутину, когда ничего не готово. Необязательный default делает его неблокирующим.

## Побочные эффекты и версионирование

Механизм маркеров, который предоставляют SDK на основе core, реализован нативно (`workflow::Context`, записывается через команды `RecordMarker` и сопоставляется при реплее):

- `ctx.SideEffect<R>(fn)` выполняет `fn` один раз, записывает её результат в историю и возвращает записанное значение при каждом последующем реплее.
- `ctx.MutableSideEffect(id, fn[, equals])` — ключевой вариант, который записывает новый маркер только при изменении значения.
- `ctx.GetVersion(change_id, min, max)` записывает номер версии при первом запуске, чтобы воркфлоу мог безопасно ветвиться на изменениях кода сквозь реплеи (возвращая `kDefaultVersion` для историй, предшествующих вызову). Процесс патчинга см. в [Версионирование](/versioning).
- `ctx.ExecuteLocalActivity<R>(opts, type, args...)` выполняет зарегистрированную активность встроенно в воркере воркфлоу, записывая её результат как маркер (повторы происходят встроенно согласно политике повторов).

## Дочерние воркфлоу

`ctx.ExecuteChildWorkflow<R>(opts, type, args...)` испускает команду `StartChildWorkflowExecution` и возвращает `Future`, который разрешается при `ChildWorkflowExecutionCompleted`/`Failed`, соотносясь напрямую по (назначенному родителем, детерминированному) `workflow_id` дочернего воркфлоу, с настраиваемой `ParentClosePolicy`. Он подключён к реплею, sticky-продолжениям, проверке детерминизма и runner ровно так же, как активности, — дочерний воркфлоу выполняется как независимый воркфлоу.

## Активности

`activity_task_handler.cpp` устроен прямолинейно: опрос → декодировать вход → найти зарегистрированную функцию по типу → выполнить её в реальном времени → `RespondActivityTaskCompleted`, либо `RespondActivityTaskFailed` при `ApplicationError`/исключении. Долгоработающие активности вызывают `RecordActivityTaskHeartbeat` (который также показывает отмену со стороны сервера). У активностей нет ограничений детерминизма. Ручки настройки политики см. в [Тайм-ауты и повторы](/timeouts-and-retries).

## Nexus-операции

Nexus реализован как полноправный тип операции. На стороне **вызывающего** `ctx.ExecuteNexusOperation<R>(endpoint, service, operation, input, schedule_to_close)` испускает команду `ScheduleNexusOperation` — как у активности, но вход и результат Nexus-операции — это каждый по **одному** значению (одному `Payload`) — и возвращает `Future`, разрешаемый событием истории `NexusOperationCompleted`/`Failed`; он проходит через реплей, sticky-продолжения и проверку детерминизма, как любая другая команда. На стороне **обработчика** `Worker::RegisterNexusOperation(service, operation, fn)` регистрирует обработчик `R Fn(Arg)`; `src/internal/nexus_task_handler.cpp` опрашивает Nexus-очередь задач, декодирует единственный входной Payload, выполняет обработчик и отвечает синхронным `StartOperationResponse`.

:::note
Сегодня обрабатываются только **синхронные** запросы start-operation — асинхронные (долгоработающие) Nexus-операции и запросы отмены возвращаются с ошибкой `NOT_IMPLEMENTED`. Текущую Nexus-поверхность см. в [матрице паритета](/parity).
:::

## Клиент

- `StartWorkflow` → `StartWorkflowExecution` (аргументы кодируются в payload-ы; UUID-идентификатор воркфлоу и request id генерируются, если не заданы).
- `WorkflowHandle::Result<R>()` длинно опрашивает `GetWorkflowExecutionHistory`, отфильтрованный до события закрытия, затем декодирует `WorkflowExecutionCompleted.result` или выбрасывает `WorkflowFailedError` для проваленных/завершённых по таймауту/прерванных/отменённых/продолженных (continued) выполнений.

## Конвертация данных

`Payload` — это `{ map<string,bytes> metadata; bytes data }`, зеркаля `temporal.api.common.v1.Payload`. `DataConverter` по умолчанию выстраивает цепочку конвертеров по ключу метаданных `encoding`: `binary/null`, `binary/plain` и `json/plain` (универсальный, через nlohmann-json). Значения пересекают API как любой сериализуемый nlohmann тип. Цепочка `PayloadCodec` (встроенные кодеки base64 и gzip; шифрование — bring-your-own, как в Go SDK) может располагаться между конвертером и проводом. См. [Конвертация данных](/data-conversion).

## Модель конкурентности

`WorkerImpl` запускает один или несколько потоков-поллеров воркфлоу, активностей и Nexus. Каждый длинный опрос использует короткий дедлайн простоя, чтобы `Stop()` возвращал управление быстро, всё ещё немедленно принимая доступные задачи. Внутри движка тело воркфлоу выполняется на собственном потоке корутины, но никогда одновременно с диспетчером (пошаговый протокол сериализует их), поэтому разделяемое состояние конкретного воркфлоу не требует внутренней блокировки.

## Сборка gRPC / protobuf

Proto-файлы `temporalio/api` вендорятся (`third_party/api`) и компилируются на этапе конфигурации (`cmake/TemporalProto.cmake`). Единственный корень импорта `-I third_party/api` разрешает каждый импорт (`temporal/…`, вендоренные `google/…`, `nexusannotations/…`); well-known типы `google/protobuf/*` берутся из системной libprotobuf, а не регенерируются заново, чтобы избежать дублирования символов. Proto-файлы `testservice` для проматывания времени компилируются из соседнего корня импорта для тестового фреймворка.

## Что намеренно отсутствует (пока)

Пункты исходной дорожной карты — обнаружение недетерминизма, версионирование воркфлоу (`GetVersion`/patching), побочные эффекты (`SideEffect`/`MutableSideEffect`), ограниченный LRU sticky-кэш, Nexus-операции, дочерние воркфлоу, повторное применение обновлений при реплее и *прерывание* при дедлоке — **все теперь реализованы** (см. разделы выше и [матрицу паритета](/parity)).

Что остаётся — это глубина, а не постатейная дорожная карта. Честные оговорки живут в [матрице паритета](/parity#roadmap), например:

- **TLS / mTLS / аутентификация по API-ключу** реализованы и проверены юнит-тестами, но ещё не запускались против настоящего Temporal-сервера с включённым TLS.
- **Nexus** обрабатывает только синхронные запросы start-operation — асинхронные операции и отмена не реализованы.
- **Сессии** поддерживают host-пиннинг + лимиты слотов, но не долгоживущие сессии с heartbeat-удержанием.
- Части поверхности **operator / версионирования воркеров** (федерация remote-cluster, часть маршрутизации) реализованы, но не полностью отработаны сквозным сценарием.

Если возможность показывает оговорку или 🟡/❌ в [матрице паритета](/parity), значит она действительно ещё не полностью на месте. Файл `docs/ARCHITECTURE.md` в репозитории (в дереве исходников) покрывает тот же дизайн на уровне файлов и символов.
