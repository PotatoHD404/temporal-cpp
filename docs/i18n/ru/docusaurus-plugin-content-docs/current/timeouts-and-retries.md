# Таймауты и повторы

Модель обнаружения сбоев в Temporal строится на простом разделении труда:
**таймауты обнаруживают сбои, повторы их смягчают.** Таймаут — это то, как сервер
замечает, что что-то пошло не так: воркер умер, задача слишком долго пролежала в
очереди, попытка зависла. Политика повторов — это то, как сервер реагирует:
планирует ещё одну попытку, с backoff, пока работа не завершится успешно или вы
не сдадитесь.

Эта страница — справочник по ручкам таймаутов и повторов, которые предоставляет
C++ SDK. Все типы опций находятся в `<temporal/common/options.h>`; heartbeat-API
на стороне активности — в `<temporal/activity/activity.h>`. О *типах сбоев*,
которые всплывают при достижении этих пределов, — `ActivityError`,
`WorkflowFailedError`, `ApplicationError` — см. [Ошибки, повторы и таймауты](/error-handling).

Примеры предполагают, что chrono-литералы находятся в области видимости:

```cpp
using namespace temporal::literals;  // 30s, 500ms, 10min, …
```

## Таймауты активностей

`ActivityOptions` несёт четыре поля таймаутов, все типа
`std::chrono::milliseconds` (присвойте любую длительность `std::chrono` — `30s`, `500ms`,
`10min` — и она преобразуется):

```cpp
struct ActivityOptions {
  std::string task_queue;                                 // по умолчанию: очередь задач воркфлоу
  std::chrono::milliseconds schedule_to_close_timeout{0}; // общий бюджет, все попытки
  std::chrono::milliseconds schedule_to_start_timeout{0}; // максимальное время в очереди до взятия в работу
  std::chrono::milliseconds start_to_close_timeout{0};    // максимальное время одной попытки (фактически обязательно)
  std::chrono::milliseconds heartbeat_timeout{0};         // максимальный интервал между heartbeat
  std::optional<RetryPolicy> retry_policy;                // не задано => поведение повторов по умолчанию сервера
};
```

Каждый измеряет свой отрезок жизни активности:

- **`start_to_close_timeout`** — максимальное время, которое может выполняться
  **одна попытка** после того, как воркер начал её исполнять. Это тот, который вы
  почти всегда задаёте. Когда попытка превышает его, сервер проваливает эту попытку
  и (согласно политике повторов) планирует следующую. В заголовке он помечен как
  *фактически обязательный*: сервер отклонит команду планирования, в которой вообще
  нет таймаута, поэтому всегда задавайте либо его, либо `schedule_to_close_timeout`.
- **`schedule_to_start_timeout`** — максимальное время, которое задача может ждать
  **в очереди задач** до того, как воркер возьмёт её в работу. Он отлавливает
  состояние «ни один здоровый воркер не опрашивает эту очередь». Задача, истёкшая по
  этому таймауту, ещё не выполнялась, поэтому её безопасно повторять даже для
  неидемпотентной работы. Оставьте его незаданным, если только у вас нет причины
  быстро падать при заторе в очереди.
- **`schedule_to_close_timeout`** — **общий** бюджет с момента, когда активность
  запланирована, до момента, когда она наконец завершается успешно или с ошибкой,
  **охватывающий все повторы**. Используйте его, чтобы поставить абсолютный потолок
  на активность, которая иначе повторялась бы долго при `start_to_close_timeout`
  × много попыток.
- **`heartbeat_timeout`** — максимальный интервал, допустимый между вызовами
  `RecordHeartbeat`. Это то, как сервер обнаруживает воркер, умерший посреди
  выполнения, **до** того, как истёк бы `start_to_close_timeout`. На практике
  обязателен для любой длительной активности (см. [Heartbeat](#heartbeating)).

```cpp
std::string ChargeCardWorkflow(temporal::workflow::Context& ctx,
                               std::string order_id, std::string amount) {
  temporal::ActivityOptions o;
  o.start_to_close_timeout    = 30s;   // каждой попытке отводится 30 с на выполнение
  o.schedule_to_start_timeout = 10s;   // быстро падаем, если ни один воркер её не берёт
  o.schedule_to_close_timeout = 5min;  // полностью сдаёмся после 5 мин повторов
  return ctx.ExecuteActivity<std::string>(o, "ChargeCard", order_id, amount).Get();
}
```

:::note
`start_to_close_timeout` ограничивает **одну попытку**; `schedule_to_close_timeout`
ограничивает **все попытки вместе**. Если вы задаёте оба, активность падает, как
только достигнут *любой* из пределов. Задание только `schedule_to_close_timeout`
также удовлетворяет правилу «таймаут обязателен», но `start_to_close_timeout`
обычно более ясный выбор, поскольку ограничивает единицу работы, которая собственно
и повторяется.
:::

## Heartbeat {#heartbeating}

Длительная активность должна периодически сообщать серверу, что она ещё жива,
вызывая `RecordHeartbeat` у своего `activity::Context`. Сочетайте его с
`heartbeat_timeout`, который короче самого длинного интервала между вызовами: если
активность перестаёт отправлять heartbeat в течение этого окна — потому что воркер
упал, завис или потерял сеть — сервер считает активность истёкшей по таймауту и
планирует повтор, вместо того чтобы дожидаться гораздо более длинного
`start_to_close_timeout`.

```cpp
temporal::ActivityOptions o;
o.start_to_close_timeout = 10min;  // один проход законно может занять минуты
o.heartbeat_timeout      = 15s;    // но он обязан отмечаться каждые 15 с
o.retry_policy = temporal::RetryPolicy{.maximum_attempts = 3};
```

`RecordHeartbeat` вариадический — передайте любые конвертируемые данные о
прогрессе **details**, и они будут переданы серверу:

```cpp
std::string ProcessLargeFile(temporal::activity::Context& ctx, std::string path) {
  const int total_chunks = CountChunks(path);
  for (int chunk = 0; chunk < total_chunks; ++chunk) {
    ProcessChunk(path, chunk);
    ctx.RecordHeartbeat(chunk);        // сообщаем о жизнеспособности + прогрессе; сбрасывает таймер heartbeat
    if (ctx.IsCancelled()) {           // сервер попросил нас остановиться
      return "cancelled";              // возвращаемся быстро и кооперативно
    }
  }
  return "done";
}
```

Два поведения делают безопасным вызов из плотного цикла:

- **Автоматическое дросселирование.** Воркер отправляет *реальный* отчёт серверу
  только когда с прошлого прошло не менее ~80% от `heartbeat_timeout` (первый вызов
  отчитывается всегда). Вызов `RecordHeartbeat` тысячи раз в секунду **не**
  затопляет сервер — на провод попадают только периодические отчёты. Интервал
  дросселирования выводится из `heartbeat_timeout` активности автоматически.
- **Отмена через heartbeat.** Отмена доставляется по каналу heartbeat, поэтому
  `IsCancelled()` отражает состояние отмены, наблюдённое последним *реальным*
  отчётом. Задросселированный вызов его не обновляет, поэтому он может отставать от
  сервера не более чем на один интервал дросселирования — это безвредно, поскольку
  следующий незадросселированный heartbeat его проявит. Опрашивайте `IsCancelled()`
  в своём цикле и быстро возвращайтесь, когда он становится истинным.

Передаваемые вами **details** о прогрессе доживают до следующей попытки: когда
активность с heartbeat истекает по таймауту и сервер её повторяет, новая попытка
может прочитать последние переданные details и продолжить, а не начинать с нуля.

:::note
Только активность с heartbeat может наблюдать отмену — `IsCancelled()` обновляется
исключительно отчётами heartbeat. Активность, которая никогда не вызывает
`RecordHeartbeat`, никогда не увидит запрос на отмену через свой контекст.
:::

## Таймауты воркфлоу

`StartWorkflowOptions` несёт три таймаута уровня воркфлоу (также
`std::chrono::milliseconds`). Они ограничивают само выполнение воркфлоу, а не
активности, которые он планирует:

```cpp
struct StartWorkflowOptions {
  std::string id;                              // по умолчанию: случайный UUID
  std::string task_queue;                      // обязательно
  std::chrono::milliseconds execution_timeout{0};
  std::chrono::milliseconds run_timeout{0};
  std::chrono::milliseconds task_timeout{0};
  std::optional<RetryPolicy> retry_policy;     // не задано => поведение повторов по умолчанию сервера
  // … memo, search_attributes, headers …
};
```

- **`execution_timeout`** — максимальная длительность **всей цепочки воркфлоу**,
  включая все повторы воркфлоу и каждое звено `ContinueAsNew`. Это абсолютный
  потолок реального времени на «этот бизнес-процесс», от начала до конца.
- **`run_timeout`** — максимальная длительность **одного запуска** (одного
  выполнения между границами continue-as-new). Без continue-as-new и без повторов
  воркфлоу запуск и выполнение совпадают.
- **`task_timeout`** — максимальное время для одной **задачи воркфлоу** (одного
  среза работы по принятию решений/реплею на воркере). Это внутренняя ручка
  тонкой настройки, которую **меняют редко**; значение по умолчанию подходит почти
  всем воркфлоу. Поднимайте его только если задача воркфлоу выполняет необычно
  тяжёлые внутрипроцессные вычисления между ожиданиями.

```cpp
temporal::StartWorkflowOptions o;
o.task_queue        = "orders";
o.execution_timeout = 24h;   // вся сага заказа должна завершиться в пределах суток
o.run_timeout       = 1h;    // любой отдельный запуск ограничен часом
auto handle = client.StartWorkflow(o, "OrderSaga", order_id);
```

Когда воркфлоу достигает одного из этих пределов, он завершается неуспешно, и
клиент видит `WorkflowFailedError` от `WorkflowHandle::Result<R>()` (с пустым
`type()`, поскольку таймаут не является прикладным сбоем). См.
[Сбой воркфлоу](/error-handling#workflow-failure).

## Политики повторов

`RetryPolicy` управляет тем, как сервер повторно предпринимает упавшую работу. То
же optional-поле `std::optional<RetryPolicy>` есть у `ActivityOptions`,
`ChildWorkflowOptions`, `LocalActivityOptions` и `StartWorkflowOptions`:

```cpp
struct RetryPolicy {
  std::chrono::milliseconds initial_interval{1000};  // задержка перед первым повтором (по умолчанию 1 с)
  double backoff_coefficient{2.0};                   // множитель интервала на каждом повторе (по умолчанию 2.0)
  std::chrono::milliseconds maximum_interval{0};     // потолок интервала; 0 => 100 × initial_interval
  int maximum_attempts{0};                           // всего попыток; 0 => без ограничений
  std::vector<std::string> non_retryable_error_types;
};
```

Задержка перед повтором *n* равна `initial_interval × backoff_coefficient^(n-1)`,
ограниченная сверху значением `maximum_interval`. Со значениями по умолчанию это
1 с, 2 с, 4 с, 8 с, …

**Оставить `retry_policy` незаданным — не то же самое, что отключить повторы.**
Незаданный optional означает *значения по умолчанию сервера*, то есть неограниченные
повторы с экспоненциальным backoff. Чтобы ограничить или изменить поведение
повторов, нужно присвоить политику:

```cpp
temporal::ActivityOptions o;
o.start_to_close_timeout = 30s;
o.retry_policy = temporal::RetryPolicy{.maximum_attempts = 3};  // не более 3 попыток
```

:::note
Синтаксис назначенных инициализаторов (`RetryPolicy{.maximum_attempts = 3}`)
задаёт только те поля, которые вы поименовали; остальные сохраняют значения по
умолчанию из структуры (`initial_interval = 1s`, `backoff_coefficient = 2.0`, …).
Это идиома, используемая во всех интеграционных тестах.
:::

Несколько распространённых форм:

```cpp
// Настроить backoff и ограничить число попыток:
o.retry_policy = temporal::RetryPolicy{
    .initial_interval    = 500ms,
    .backoff_coefficient = 1.5,
    .maximum_interval    = 30s,
    .maximum_attempts    = 5,
};

// Ровно одна попытка — быстро падаем, без повторов:
o.retry_policy = temporal::RetryPolicy{.maximum_attempts = 1};

// Никогда не повторять эти типы сбоев, независимо от maximum_attempts:
o.retry_policy = temporal::RetryPolicy{
    .non_retryable_error_types = {"CardDeclined", "InvalidInput"},
};
```

### Остановка повторов изнутри активности

Есть два способа, которыми попытка активности может объявить сбой постоянным, чтобы
сервер немедленно отказался от неё вместо планирования дальнейших попыток:

1. **Бросить неповторяемый `ApplicationError`** — задайте третий аргумент
   конструктора:

   ```cpp
   // Внутри активности:
   throw temporal::ApplicationError("card declined — do not retry",
                                    "CardDeclined", /*non_retryable=*/true);
   ```

2. **Перечислить тип ошибки в `non_retryable_error_types`** — если строка типа
   брошенного `ApplicationError` присутствует в этом списке, активность считается
   неповторяемой, даже когда флаг конструктора равен `false` и `maximum_attempts`
   ещё не достигнуто.

В любом случае цикл повторов останавливается на этой попытке. Любое другое
брошенное исключение повторяемо по умолчанию и будет предпринято снова, пока
политика не исчерпается. Полный конструктор см. в
[Генерация сбоев из активности](/error-handling#throwing-failures-from-an-activity).

## Как сбой доходит до воркфлоу

Повторы и таймауты обрабатываются сервером, прозрачно для воркфлоу — воркфлоу не
видит промежуточных упавших попыток. Он наблюдает только итог, когда ожидает
результат активности. Как только повторы исчерпаны (или `maximum_attempts` было
равно 1, или была брошена неповторяемая ошибка), вызов `Future<R>::Get()`
бросает `temporal::ActivityError`:

```cpp
temporal::ActivityOptions o;
o.start_to_close_timeout = 30s;
o.heartbeat_timeout      = 15s;
o.retry_policy = temporal::RetryPolicy{.maximum_attempts = 3};

try {
  std::string result = ctx.ExecuteActivity<std::string>(o, "ProcessLargeFile", path).Get();
  // сбои каждой попытки были поглощены повторами; этот выполнился успешно
} catch (const temporal::ActivityError& e) {
  // повторы исчерпаны, неповторяемая ошибка или таймаут
  if (e.type().find("Timeout") != std::string::npos) {
    ctx.GetLogger().Warn("activity timed out", {{"type", e.type()}});
  }
  throw;  // перебрасываем, чтобы провалить воркфлоу, либо обрабатываем корректно
}
```

`ActivityError::type()` повторяет тип сбоя. Для таймаута это один из
`"TimeoutType_START_TO_CLOSE"`, `"TimeoutType_SCHEDULE_TO_CLOSE"` или
`"TimeoutType_HEARTBEAT"`; для прикладного сбоя это строка типа, которую вы
передали в `ApplicationError`. Сопоставление с ней описано в
[Перехват сбоев активности в воркфлоу](/error-handling#catching-activity-failures-in-a-workflow).

## Выбор таймаутов

Практическая отправная точка:

- **Всегда задавайте `start_to_close_timeout`** как щедрую, но конечную границу
  одной попытки активности — достаточно длинную, чтобы здоровый запуск никогда её
  не задевал, достаточно короткую, чтобы зависший воркер был замечен за разумное
  время.
- **Добавляйте `heartbeat_timeout` (и heartbeat) для любой активности, которая
  может выполняться дольше нескольких секунд.** Это единственный таймаут, который
  обнаруживает мёртвый воркер *во время* попытки; без него вы дожидаетесь полного
  `start_to_close_timeout`.
- **Используйте `schedule_to_close_timeout`**, когда вам нужен абсолютный потолок
  от начала до конца поверх повторов, например «это должно завершиться успешно в
  пределах 10 минут или эскалировать».
- **Тянитесь к `schedule_to_start_timeout`** только чтобы быстро падать при заторе
  в очереди; повторять безопасно, поскольку задача ещё не начала выполняться.
- **Оставляйте `task_timeout` (и обычно `run_timeout`) на значении по умолчанию.**
  Задавайте `execution_timeout`, чтобы ограничить весь бизнес-процесс, когда таковой
  применим.
- **Ограничивайте повторы осознанно.** По умолчанию сервер не ограничивает их;
  присвойте `RetryPolicy` с `maximum_attempts` (и пометьте действительно постоянные
  сбои неповторяемыми), чтобы обречённая активность не повторялась вечно.

См. [матрицу паритета](/parity) для полного статуса возможностей и
[Ошибки, повторы и таймауты](/error-handling) для типов сбоев, которые порождают
эти пределы.
